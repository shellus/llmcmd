# Vertex AI Gemini 能力与请求方式总览

## 目的

本文档记录两类事实：

1. Vertex AI / Gemini 官方文档声明的模型输入输出能力
2. `shellus-llmcmd` 当前代码实际采用的附件提交方式与请求格式

本文档用于为后续的图片、音频、视频附件能力维护提供可追溯依据，避免把模型能力、网关能力与本程序当前实现混为一谈。

## 范围

本次记录聚焦以下模型与能力：

- 理解模型：`gemini-2.5-flash`、`gemini-3-flash`、`gemini-3.1-flash-lite`
- 图片输出模型：`gemini-2.5-flash-image`、`gemini-3.1-flash-image`
- 音频输出模型：`gemini-3.1-flash-tts-preview`
- 补充说明：`gemini-2.5-flash-live-api` 具备音频输出，但不属于本项目当前 `tts` 实现路径

## 命名结论

本次检索到的官方标准页面包含：

- `Gemini 2.5 Flash`
- `Gemini 3 Flash`
- `Gemini 3.1 Flash-Lite`
- `Gemini 2.5 Flash Image`
- `Gemini 3.1 Flash Image`
- `Gemini 3.1 Flash TTS Preview`

在本次官方检索范围内，**未检索到**单独命名为 **`Gemini 3.1 Flash`** 的标准模型页。因此后续文档与代码注释不应把 `gemini-3.1-flash` 当成已确认的官方标准模型名。

## 实施前结论

### 1. 视频理解不是“先手工拆帧再发图”

Gemini 官方文档明确把视频列为独立输入类型，支持：

- 直接上传视频文件
- File API
- Cloud Storage / GCS URI
- YouTube URL
- 小文件 inline data

因此，后续为本项目增加视频理解支持时，应按“直接提交视频输入”建模；模型内部是否抽帧属于模型实现细节，不应在 CLI 层强制改写成“图片序列输入”。

### 2. 当前程序已实现第一阶段视频理解附件输入

当前 `shellus-llmcmd` 已实现：

- `chat -r` 的图片 / 文本 / 音频 / 视频附件理解
- `image -r` 的图片 / 文件参考输入
- `video -r` 的首帧参考图上传

当前 **仍未实现**：

- `chat --edit -r` 的视频附件辅助编辑
- `image -r` 直接提交视频文件
- Gemini 原生 `file_data` / `file_uri` 的视频专用请求分支

### 3. 图片输出与音频输出应按模型类别区分

- 通用理解模型的官方模型页均写明 **输出为 Text**
- 图片输出能力应仅记录在 `-image` 后缀模型上
- 音频输出能力应记录在 TTS 或 Live API 模型上，不能反推为所有 Gemini 模型都支持音频输出

| 结论 | 主要来源 | 状态 |
|------|----------|------|
| `gemini-2.5-flash` 支持图/音/视频输入，输出文本 | Vertex `2-5-flash` 模型页；Gemini API `gemini-2.5-flash` 模型页 | 已记录 |
| `gemini-3-flash` 支持图/音/视频输入，输出文本 | Vertex `3-flash` 模型页；Vertex 图/音/视频理解页 | 已记录 |
| `gemini-3.1-flash-lite` 支持图/音/视频输入，输出文本 | Vertex `3-1-flash-lite` 模型页；Vertex 图/音/视频理解页 | 已记录 |
| 视频理解支持直接视频文件/File API/GCS/YouTube | Gemini API `video-understanding`；Vertex `video-understanding` | 已记录 |
| `-image` 模型支持图片输出 | Vertex `2-5-flash-image`、`3-1-flash-image` 模型页 | 已记录 |
| `gemini-3.1-flash-tts-preview` 支持音频输出 | Gemini API `gemini-3.1-flash-tts-preview` 模型页；Gemini API `speech-generation` | 已记录 |
| `gemini-3.1-flash` 是标准官方模型名 | 本次官方检索未找到对应模型页 | 未证实 |
